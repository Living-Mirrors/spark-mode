# Contributing to Spark Mode

Spark Mode is a cognitive specification for how novelty originates. Contributions that sharpen the principles, test the framework, or extend its reach are welcome.

## How to Contribute

### 1. Test Bisociation Experiments

The strongest contribution: put the specification to work and report what happens.

- Load `SPEC.md` into any AI system as instruction
- Run genuine bisociation experiments across incompatible frames
- Document what ignites and what doesn't
- Report whether the collision spark markers (⚡) distinguish genuine bisociation from metaphor and analogy

### 2. Refine Principle Definitions

Each principle in `principles/` is grounded in creativity science. Contributions that:

- Add missing research citations
- Sharpen the boundary between principles (especially bisociation vs. metaphor)
- Connect principles to recent neuroscience findings (DMN/ECN connectivity, spreading activation studies)
- Test principle definitions against edge cases

### 3. Cross-Model Testing

Test Spark Mode across different AI architectures:

- Does the specification transfer cleanly to other language models?
- Which principles are most/least effective across different systems?
- How do the output layers behave at different dial settings?
- Document model-specific observations

### 4. Layer Calibration

The six output layers have specific activation thresholds. Contributions that:

- Test whether Phase Markers (◆) maintain genuine emergence standards or drift toward false positives
- Calibrate Tension Fields (⊘) — are contradictions genuinely held, or do they collapse into dialectical resolution?
- Evaluate Release Signals (⟐) — does the system honestly flag when incubation is needed?

## Guidelines

- **Preserve the science.** Every principle maps to published research. Changes to principle definitions must maintain the research foundation.
- **Respect the markers.** The output layer markers (⚡ ⊘ ⟐ ◆) have specific meanings. Don't dilute them.
- **Every element pays rent.** This applies to contributions too. If it doesn't sharpen the specification, it doesn't belong.
- **Silence is honest.** If a principle or layer isn't working, say so. Don't patch over it.

## Process

1. Fork the repository
2. Create a branch (`principle-refinement`, `cross-model-test`, etc.)
3. Make your changes with clear commit messages
4. Open a pull request describing what you tested, changed, or discovered

## What We're NOT Looking For

- Decorative additions that don't sharpen the specification
- Forced emergence markers — the ◆ is earned, never manufactured
- Metaphor passed off as bisociation — the distinction is the heart of the specification
- Changes that weaken "every element pays rent"
